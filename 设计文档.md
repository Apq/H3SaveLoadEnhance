# SaveLoadEnhance 设计文档

## 概述

玩家手动存档或读档后，插件在进程内记住该存档文件名。下次打开存档/读档界面时，自动选中该文件所在行并滚动到可见。

状态仅存内存，不写文件，重启清空。插件启停由 HD Mod 插件列表控制。

## 架构：5 个 Hook

| Hook | 地址 | 类型 | 职责 |
|------|------|------|------|
| `HH_SaveGame` | 0x4BEB60 | HiHook SPLICE_ THISCALL_ | 辅助记录：存档界面可见时调用 SaveGame → 记录 |
| `HH_LoadSaveConfirm` | 0x58BFB0 | HiHook SPLICE_ FASTCALL_ | 捕获候选：读档窗口确认处理开始时读取当前选中文件名 |
| `Hook_AfterManualLoadGame` | 0x58C62E | LoHook | 确认记录：手动读档路径中 `LoadGame` 返回后读取 EAX，成功才记录候选文件名 |
| `Hook_DialogShow` | 0x584EF4 | LoHook | 每次对话框显示时自动选中记录文件 |
| `HH_DialogDestructor` | 0x583E10 | HiHook SPLICE_ THISCALL_ | 读档对话框关闭不记录；存档对话框关闭只等待 SaveGame 确认 |

### HH_SaveGame（0x4BEB60）

H3Main::SaveGame 入口。HiHook SPLICE_ THISCALL_ 包裹，签名 `(HiHook*, DWORD this, const char* save_path, DWORD a3, DWORD a4, DWORD a5, DWORD a6)`。

**必须传递全部 6 个参数**（this + save_path + 4 个固定值 `1,1,1,0`）。

记录机制：`g_save_dialog_visible` 为 true，或存档对话框刚关闭后的短暂窗口期内，记录 `SaveGame` 传入的文件名。这样可以兼容「先析构对话框 → 再调用 SaveGame」的执行顺序，同时避免用户只是关闭/取消存档窗口时误记录。

### HH_LoadSaveConfirm（0x58BFB0）

读档/存档对话框确认处理函数。HiHook SPLICE_ FASTCALL_ 包裹，签名 `(HiHook*, char* self)`，对应反编译 `uint __fastcall FUN_0058bfb0(int param_1)`。

记录机制：进入原函数前从读档对话框对象读取当前选中文件名，写入 `g_pending_manual_load_name`，并设置 `g_pending_manual_load`。该 hook 不直接记录读档，因为 `FUN_0058bfb0` 自身返回值不是底层 `LoadGame` 返回值。

不再 hook 底层 `H3Main::LoadGame` 函数入口（0x4BEFF0）。该入口会被新游戏、战役、初始化、回退等非手动路径调用，且实测在读档流程中包裹它可能导致卡死。

### Hook_AfterManualLoadGame（0x58C62E）

位于手动读档路径 `FUN_0058c560` 内部，紧跟 `call 0x4BEFF0` 之后。该位置是干净指令边界：`0x58C629 call 0x4BEFF0`，`0x58C62E mov ecx, [ebp-4]`。

LoHook 在此读取 `HookContext::eax`，此时 EAX 仍是 `H3Main::LoadGame` 返回值。只有 `EAX != 0` 且存在 `g_pending_manual_load_name` 时，才记录为“手动读档”。如果 `EAX == 0`，写日志并清理候选，不更新最后记录。

### Hook_DialogShow（0x584EF4）

位于 `fcn.00584EC0`（对话框显示/消息循环入口）内部，`call fcn.00602D60`（消息循环）之前。

- **保持 LoHook**：hook 点在函数内部（非函数边界），HiHook 无法替代
- 当前地址是干净指令边界（`mov ecx, [0x6992D0]`，绝对寻址），LoHook 在此稳定可靠
- **每次对话框显示都会触发**，不管窗口是新建还是复用（连续存档）
- ESI = 对话框对象，读 `[ESI+0x64]`/`[ESI+0x65]` 区分界面类型
- 读档/存档界面且有记录时：调用 `ApplyRecordedSelection` 自动选中
- 设置 `g_save_dialog_visible`（存档界面时为 true）
- 打开读档界面时清理旧的读档候选，不记录任何文件名；记录只发生在 `Hook_AfterManualLoadGame` 看到 `LoadGame` 成功返回之后

**存档界面特殊处理**：存档界面有文本输入框控件，在消息循环前调用 `UpdateForSelectedScenario` 或 `Redraw` 会破坏输入框初始化。因此存档界面只设置 `selected` 索引和 `topIndex` 到内存字段，让原生消息循环自行处理。

### HH_DialogDestructor（0x583E10）

对话框析构函数入口。HiHook SPLICE_ THISCALL_ 包裹，签名 `(HiHook*, char* self, int delete_flag)`。

**必须传递隐藏的 `delete_flag` 参数**（MSVC C++ 析构函数：0=仅析构，1=析构+delete）。

读档界面关闭时不直接记录。用户可能只是关闭或取消窗口，必须等 `Hook_AfterManualLoadGame` 看到 `LoadGame` 成功返回后才记录。

存档界面关闭时不直接记录当前选中项，因为用户可能只是取消/关闭，并没有实际保存。此时只设置一个短暂的 pending 状态，后续如果 `SaveGame` 真正被调用，再由 `HH_SaveGame` 记录文件名。

先读取数据再调用原函数（析构会释放内部资源）。

## 自动选中逻辑（ApplyRecordedSelection）

1. 在显示数组中按文件名查找目标索引
2. 设置 `selected`（+0x374）和 `topIndex`（+0x370）
3. 同步滚动条对象（+0x183C）的 tick 值
4. 读档界面：调用 `UpdateForSelectedScenario`（0x5857D0）+ `Redraw`（0x584820）刷新 UI
5. 存档界面：跳过步骤 4（避免破坏输入框）

## 界面区分

| [+0x64] | [+0x65] | 界面 |
|---------|---------|------|
| 1 | 0 | 读档 |
| 0 | 1 | 存档 |
| 0 | 0 | 新游戏选图 |

## 日志格式

每行带时间戳。术语：**记录** = 写入内存文件名，**更新** = 修改界面选中项。

## 对话框对象关键字段

| 偏移 | 字段 | 说明 |
|------|------|------|
| +0x64 | 模式标志 | 1=读档 |
| +0x65 | 模式标志 | 1=存档 |
| +0x370 | topIndex | 列表顶部索引 |
| +0x374 | selected | 当前选中索引 |
| +0x1054 | 显示数组起始指针 | begin |
| +0x1058 | 显示数组结束指针 | end |
| +0x183C | 滚动条对象指针 | — |

显示数组单项步长 `0xCA4`，文件名偏移 `0x33D`。可见行数：读档 18 行，存档 16 行。
